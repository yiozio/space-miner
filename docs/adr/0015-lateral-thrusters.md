# 0015. 横向きスラスタとストラフ操作

## Status
Accepted (ADR 0006 の「横向きスラスタ無効」条項を更新する)

## Context
- ADR 0006 ではスラスタ回転 R=1, R=3 を `Sideways` として一括し、推進にも炎にも寄与しない仕様としていた。
- 機軸転換だけで横移動を作ると、海賊との戦闘や狭い小惑星帯での回避で機体が大きく回転する必要があり、戦闘の機微を出しづらかった。
- 既にエディタ・描画・スラスタ集計の枠組みは Rotation を意識して作られていたため、横向きスラスタを「機能する向き」として活かす余地があった。

## Decision

### 横向きを 2 方向に分割
- `ThrustDir` を `Forward (R=0) / Right (R=1) / Backward (R=2) / Left (R=3)` の 4 値に拡張。
- 旧 `ThrustDirSideways` は廃止。
- ebiten の `GeoM.Rotate(R*π/2)` は `(x, y) → (-y, x)` の写像で画像の噴射口を視覚的に CW 90°×R 回転させる。
  推進はその噴射方向の反作用なので、R=1 では「噴射が機体左 → 推進が機体右」、R=3 では「噴射が機体右 → 推進が機体左」となる。
- アフターバーナー描画 (`drawAfterburners`) と推進方向のロジックを同じ回転規則で揃え、画像の噴射口・炎の向き・実際の加速方向が一致するようにした
  （旧実装は R=1/R=3 で `rxL, ryL` の符号がずれており、噴射口の絵と炎の出る側が逆になっていた）。

### 入力と推進
- `Q` で左ストラフ、`E` で右ストラフ。前後 (W/S) と独立に評価し、同時押しで斜め推進になる。
- 同方向で重複する入力 (Q+E、W+S) はそれぞれ片方を優先する (Q 優先・W 優先)。
- ブースト (Shift) は方向ごとに独立適用。`Fuel > 0` のときに対応する `BoostFuelCost` を消費する。

### 速度上限
- `Player.fwdCap / bckCap / lftCap / rgtCap` の 4 つに拡張。
- 速度マグニチュードは「現在稼働中の方向 cap の最大値」にクランプする。何も稼働していなければ 4 cap の最大値。
- これにより、前進だけしているときに横ストラフ用の速い cap でブーストし続けてしまう過大評価を防ぐ。

### 描画
- `ThrustActiveDir` をビットフラグ化 (`Forward=1 / Backward=2 / Left=4 / Right=8`) し、複数方向を同時に表現できるようにした。
- `Ship.thrustEmitters()` は稼働中ビットに対応する Rotation のスラスタをすべて返す。炎方向の導出ロジック (Rotation → 後端ベクトル) は ADR 0006 のものをそのまま流用。
- 非常用 Cockpit 推進は引き続き前方のみ。Q/E では Cockpit フォールバックは効かない (横スラスタを積まないとストラフ不可)。

## Consequences
- 戦闘で「機体を回さずに横へ逃げる」「敵から距離を保ったまま射線を維持する」など、空間操作の自由度が増した。
- 横スラスタの装備・燃料コストが新しい意思決定対象になり、ビルドの幅が広がる。
- ADR 0006 の「Sideways は無意味」記述は本 ADR で置き換えられる。0006 自体は前後スラスタ集計と動的キャップの基本ロジックに関しては引き続き有効。
- セーブには `ThrustActiveDir` を含めないため、ビットフラグ化による互換性影響はない。
